## 一、项目概述

本项目是一个**基于大语言模型（LLM）的智能八字命理分析系统**，核心探索"传统领域知识 + 大语言模型"的深度融合范式。系统由两个子系统构成：

1. **符号计算层**（传统 AI 的专家系统思想）：纯 Python 实现的八字排盘引擎，将传统命理学的排盘规则（节气计算、干支推算、十神判定、大运排列等）编码为确定性算法
2. **神经网络推理层**（现代 LLM 应用）：调用 DeepSeek 大语言模型，通过精心设计的 Agent System Prompt（约 74,000 字符），将 30 年命理师的专业知识体系注入模型，实现对命盘的 7 级递进式智能分析

**核心研究问题**：如何通过 Prompt Engineering 将非结构化领域专家知识转化为 LLM 可稳定执行的推理链路？

### 2.2 设计理念：符号主义 + 连接主义的混合架构

本项目体现了 AI 领域两种范式的互补：

| 维度 | 符号计算层 | LLM 推理层 |
|------|-----------|-----------|
| **AI 范式** | 符号主义（专家系统） | 连接主义（大语言模型） |
| **知识表示** | 显式规则（天文公式、查表、口诀） | 隐式知识（嵌入在模型参数中） |
| **推理方式** | 确定性演绎推理 | 概率性归纳推理 |
| **优势** | 精确、可验证、可解释 | 灵活、能处理模糊语义、有"常识" |
| **局限** | 无法处理模糊判断 | 可能产生幻觉、结论不一致 |
| **本系统角色** | 精确计算四柱/大运/十神等结构化数据 | 基于结构化数据进行格局判定、旺衰分析、人生建议 |

## 三、LLM 应用核心技术

### 3.1 Agent System Prompt 设计（Prompt Engineering）

这是本系统最核心的 AI 工程工作。Agent 定义文件（`traditional-bazi-master.md`）约 **74,000 字符（933 行）**，是一份高度结构化的领域知识注入文档。

**设计原则**：

#### （1）人设锚定 — 角色一致性

```
你是一位拥有30年实战经验的传统八字命理师。
你师承门派正统一脉，深研《渊海子平》《三命通会》
《滴天髓》《穷通宝鉴》《子平真诠》《神峰通考》
等古籍经典，经手过数千命例……
```

通过明确的角色设定，约束模型的输出风格、知识边界和语气基调，减少"AI 味"。

#### （2）推理链设计 — 7 级递进分析

系统将命理分析拆解为不可跳步的 7 级推理链：

```
格局定框架 → 旺衰定喜忌 → 调候定顺逆
    → 十神定人事 → 刑冲合害定细节
    → 神煞为辅助 → 大运流年定应期
```

这是典型的 **Chain-of-Thought（思维链）** 设计——不直接让模型"分析这个八字"，而是强制模型按优先级逐层推理，每层输出后再进入下一层。这显著提升了分析的逻辑一致性。

#### （3）知识注入 — 31 个核心概念的精确定义

System Prompt 包含一个完整的"核心概念库"，对 31 个命理学术语给出了严格的操作性定义。例如：

```markdown
**14. 身强**
- 严格定义：日主综合得令、得地、得势后力量充足，能胜任财官
- 区分要点：身强宜克泄耗（官杀、食伤、财星），忌生扶（比劫、印星）
- 实战示例：甲木生于寅月（得令），地支又见卯木（得地），
  天干有比劫（得势），身强无疑
```

每个概念包含：**严格定义 → 区分要点 → 实战示例**。这种结构化知识表示确保了模型在推理时使用统一的术语体系，避免概念混淆导致的推理错误。

#### （4）约束边界 — 负面清单

```
### 严格禁止的行为
- 不妄称可以靠某种方法百分百改变命运
- 不推销任何付费化解法事或风水物品
- 不妄断生死精确日期
- 不对用户做道德审判
```

通过明确禁止特定行为，降低模型产生不当输出的风险。

#### （5）结构化输出格式

System Prompt 规定了标准批命格式（5 个部分），其中"事业财运"和"婚姻感情"各有 6 个必输出子项。这种输出规范保证了分析报告的完整性和一致性。

### 3.2 用户消息构造（Message Construction）

`analysis_service.py` 中的 `_build_user_message()` 函数将排盘数据转换为结构化的 Markdown 分析请求：

```
┌─────────────────────────────────────┐
│ 1. 基础信息（公历/农历/性别/经纬度）  │
│ 2. 四柱干支表（含十神/纳音/藏干/长生）│
│ 3. 核心数据（日主/起运/空亡/胎元）   │
│ 4. 大运排盘表（8步完整表格）         │
│ 5. 10 章分析要求 + 排版格式要求       │
└─────────────────────────────────────┘
```

**关键设计决策**：
- 使用 Markdown 表格而非 JSON 传递命盘数据，因为 LLM 对 Markdown 表格的理解能力远优于 JSON
- 分析要求明确到章节级别，约束输出结构
- 单次分析 `max_tokens: 24576`，`temperature: 0`（需要严谨推理，不需要创意发散）
- 续接对话 `max_tokens: 8192`，降低 token 消耗

### 3.3 多轮对话管理（Conversation Continuation）

系统支持"续接分析"——当 Agent 在验盘阶段需要用户确认过去事件时，用户可以回复，系统会携带完整 `messages` 数组继续对话：

```python
# 初始分析
messages = [
    {"role": "system", "content": system_prompt},
    {"role": "user", "content": user_message},
    {"role": "assistant", "content": analysis_text},
]

# 续接
api_messages = messages[1:]  # 历史对话（不含 system）
api_messages.append({"role": "user", "content": user_reply})
# system prompt 单独传递
payload = {"system": messages[0]["content"], "messages": api_messages}
```

这遵循了 Anthropic Messages API 的 conversation 范式，system prompt 在每轮中保持一致。

### 3.4 API 管理与限流

| 机制 | 实现 |
|------|------|
| **配置优先级** | 环境变量 → `~/.claude/settings.json` → `config.local.py`（适配不同部署环境） |
| **限流** | IP 级别滑动窗口，深度分析 3 次/时，续接 5 次/时（缓存命中不消耗配额） |
| **超时** | 单次分析 120 秒，续接对话 180 秒 |
| **Token 追踪** | 返回 `usage` 对象（prompt_tokens / completion_tokens） |
| **密码保护** | 可选 `WEB_PASSWORD` 配置，输错 5 次锁定 3 分钟 |
| **分析缓存** | 服务端基于四柱+性别+起运生成缓存键，最大 50 条，1 小时 TTL |

## 四、符号计算层（传统 AI 方法）

在 LLM 兴起之前，八字排盘只能通过专家系统实现。本系统的符号计算层本质是一个**确定性规则引擎**。

### 4.1 双引擎策略（鲁棒性设计）

```
优先级 1：sxtwl（C++ 寿星天文历）
    ├─ 精度：±1 秒
    └─ 条件：需编译安装

优先级 2：纯 Python 回退（自动启用）
    ├─ 节气：Meeus 天文公式计算太阳视黄经
    ├─ 精度：~0.001°（起运差异 ~0.1 岁）
    ├─ 优化：3年×24节气预计算缓存
    ├─ 依赖：仅 math 模块（零外部依赖）
    └─ 包含：从 solar 到 lunar 的完整回退实现
```

### 4.2 核心算法

| 算法 | 方法 | 形式化程度 |
|------|------|-----------|
| 四柱干支 | 查表法 + 五虎遁/五鼠遁口诀编码 | 完全形式化 |
| 十神关系 | 五行生克矩阵 + 阴阳异同判定 | 完全形式化 |
| 起运时间 | 节气日差 ÷ 3 = 起运年龄 | 完全形式化 |
| 大运排列 | 月柱干支步进（顺/逆行各进/退一位） | 完全形式化 |
| 纳音 | 60 甲子查表 | 完全形式化 |
| 空亡 | 旬空数学判定 | 完全形式化 |
| 十二长生 | 阳顺阴逆查表 | 完全形式化 |
| 夜子时处理 | 23:00-23:59 日柱按次日、时柱按次日日干 | 完全形式化 |

### 4.3 与 LLM 层的分工边界

**符号计算层负责**：所有可形式化、可查表、可公式化的计算。
**LLM 层负责**：需要模糊判断、经验推理、语境理解的"断命"环节。

这种分工体现了 AI 系统的经典设计原则：**让确定性算法做确定性的事，让概率模型做概率性的事。**

## 五、关键技术挑战与解决方案

### 5.1 LLM 推理一致性问题

**挑战**：同一命盘多次分析可能得出不同结论（这是 LLM 概率性的本质）。

**解决方案**：
- `temperature: 0` — 最小化随机性
- 7 级递进推理链 — 约束推理路径
- 31 个概念精确定义 — 统一术语体系
- 必输出子项清单 — 防止遗漏关键维度
- **服务端分析缓存** — 同命盘命中缓存后秒返，避免重复分析导致结论不一致

### 5.2 领域知识注入效率

**挑战**：如何把 30 年命理师经验高效注入 LLM？

**解决方案**：
- 不使用 Few-shot（示例会占据大量 token 且覆盖不足）
- 使用"方法论注入"——教模型"怎么分析"而非"分析什么结论"
- 每个概念用"定义 + 区分 + 示例"三元组表达，兼顾准确性和可操作性
- System Prompt 达 ~74K 字符，提供足够丰富的领域知识上下文

### 5.3 幻觉抑制

**挑战**：LLM 可能编造不存在的古籍引用或杜撰命理术语。

**解决方案**：
- System Prompt 明确要求"引用经典出处"
- 要求"不确定处标明【待验证】"
- 验盘机制：先反推已知事件验证命盘准确性，再批断未来

### 5.4 长耗时分析的断线恢复

**挑战**：深度分析耗时 4-6 分钟，期间用户切标签页或关闭页面会导致 HTTP 请求中断，浪费已推理的 token。

**解决方案（三层防护）**：

| 层级 | 机制 | 说明 |
|------|------|------|
| **服务端缓存** | 四柱+性别+起运作哈希键，50 条/1 小时 TTL | 命中后秒返，不消耗限流配额 |
| **客户端 pending** | `bazi_analysis_pending` 存入 localStorage | 页面加载时检测未过期 pending 自动重试 |
| **运行时恢复** | `visibilitychange` + AbortError 自动重试 | 切回标签页自动恢复，指数退避（5s/15s/30s，3 次） |

此外，分析开始前设置 `beforeunload` 事件提醒用户，8 分钟超时通过 AbortController 自动取消。

### 5.5 部署环境适配

**挑战**：PythonAnywhere 免费账户无环境变量、无法下载文件、WSGI 进程管理特殊。

**解决方案**：多级配置回退（环境变量 → `settings.json` → `config.local.py`），静态资源本地化，禁止 CSP 头和外部 CDN。

### 5.6 跨平台中文字体兼容

**挑战**：PDF 生成需要 CJK 字体，但 Linux/macOS/Windows 字体路径各不相同。

**解决方案**：`generate_bazi_pdf.py` 中实现 `_find_cjk_font()` 函数，按优先级自动探测各平台字体路径，并内置 `fonts/` 目录作为回退。

## 十、不足与改进方向

1. **推理可解释性**：LLM 分析为黑盒输出，可引入"分步输出"让用户看到每层推理的中间结果
2. **知识库外挂（RAG）**：可将古籍原文向量化，分析时检索相关经典段落作为引用依据
3. **一致性验证**：缺少对同一命盘多次分析结论一致性的自动化评估
4. **流式输出**：当前为同步返回，4-6 分钟等待体验较差，可改为 SSE 流式输出
5. **Agent 自进化**：验盘反馈（`feedback_logs/`）可用于微调或优化 System Prompt
6. **评估基准**：缺乏对分析质量的量化评估指标
7. **缓存策略优化**：当前为简单 LRU + TTL，可引入持久化缓存（SQLite/Redis），跨进程重启保留
8. **图表交互性**：当前 SVG 为静态渲染，可增加鼠标悬浮高亮、点击展开等交互

## 十一、总结

本项目实践了"**传统领域知识 × 大语言模型**"的混合 AI 架构。核心创新在于：

1. **用确定性算法（符号计算层）保证底层数据的准确性**——八字排盘不允许有误差
2. **用精心设计的 System Prompt（~74K 字符）将领域专家的推理方法论注入 LLM**——不是教模型"答案是什么"，而是教它"怎么思考"
3. **用 7 级递进推理链约束模型的推理路径**——强制 CoT，避免跳步推理导致的逻辑混乱
4. **用验盘机制形成人机协作闭环**——模型反推过去事件，用户验证，反馈后模型调整并继续
5. **用三层断线恢复机制提升长耗时分析的可靠性**——服务端缓存 + 客户端持久化 + 运行时自动重试，确保 4-6 分钟分析不丢失
6. **用反馈日志系统积累验盘数据**——每次分析自动保存，为后续 Agent 优化提供数据支撑

这为"AI + 传统文化"领域提供了一个可复用的技术范式：**符号计算做"硬"的部分（可形式化的规则），LLM 做"软"的部分（需要经验判断的模糊推理），两者通过结构化数据接口耦合。**
