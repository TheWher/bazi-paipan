## 一、项目概述

本项目是一个**基于大语言模型（LLM）的智能命理分析系统**，核心探索"传统领域知识 + 大语言模型"的深度融合范式。系统由两个子系统构成：

1. **符号计算层**（传统 AI 的专家系统思想）：纯 Python 实现的排盘引擎，将传统命理学的排盘规则（节气计算、干支推算、十神判定、大运排列等）编码为确定性算法
2. **神经网络推理层**（现代 LLM 应用）：调用 DeepSeek 大语言模型，通过精心设计的 Agent System Prompt（约 123,000 字符），将 30 年命理师的专业知识体系注入模型，实现对命盘的 9 级递进式智能分析

**核心研究问题**：如何通过 Prompt Engineering 将非结构化领域专家知识转化为 LLM 可稳定执行的推理链路？

### 1.1 系统扩展：双命理体系

2026 年 7 月，系统从单一八字（Ba Zi）扩展为八字 + 紫微斗数（Zi Wei Dou Shu）双命理体系：

| 维度 | 八字 | 紫微斗数 |
|------|------|----------|

| **方法论体系** | 梁湘润体系 | 三合中州派 |
| **知识库 JSON** | 8 个文件（30KB） | 8 个文件（146KB） |

### 2.2 设计理念：符号主义 + 连接主义的混合架构

本项目体现了 AI 领域两种范式的互补：

| 维度 | 符号计算层 | LLM 推理层 |
|------|-----------|-----------|
| **AI 范式** | 符号主义（专家系统） | 连接主义（大语言模型） |
| **知识表示** | 显式规则（天文公式、查表、口诀） | 隐式知识（嵌入在模型参数中） |
| **推理方式** | 确定性演绎推理 | 概率性归纳推理 |
| **优势** | 精确、可验证、可解释 | 灵活、能处理模糊语义、有"常识" |
| **局限** | 无法处理模糊判断 | 可能产生幻觉、结论不一致 |
| **本系统角色** | 精确计算四柱/大运/十神等结构化数据 | 基于结构化数据进行格局判定、旺衰分析、人生建议 |

## 三、LLM 应用核心技术

### 3.1 Agent System Prompt 设计（Prompt Engineering）

这是本系统最核心的 AI 工程工作。八字 Agent 定义文件（`traditional-bazi-master.md`）约 **123,000 字符（1508 行）**，是一份高度结构化的领域知识注入文档。紫微斗数 Agent（`ziwei-master.md`）约 **49,000 字符（763 行）**。

**设计原则**：

#### （1）人设锚定 — 角色一致性

```
你是一位拥有30年实战经验的传统八字命理师。
你师承门派正统一脉，深研《渊海子平》《三命通会》
《滴天髓》《穷通宝鉴》《子平真诠》《神峰通考》
等古籍经典，经手过数千命例……
```

通过明确的角色设定，约束模型的输出风格、知识边界和语气基调，减少"AI 味"。

#### （2）推理链设计 — 9 级递进分析（vs 最初 7 级）

系统将命理分析拆解为不可跳步的 9 级推理链：

```
调候定顺逆 → 格局定框架 → 旺衰定喜忌
    → 病药定方向 → 流通定气象 → 十神定人事
    → 刑冲合害定细节 → 神煞为辅助 → 大运流年定应期
```

这是典型的 **Chain-of-Thought（思维链）** 设计——不直接让模型"分析这个八字"，而是强制模型按优先级逐层推理，每层输出后再进入下一层。从最初 7 级演进到 9 级，增加了"病药"和"流通"两个中间层，因为实测发现跳过这两层直接进入十神分析会导致调候结论与十神判断矛盾。

**紫微斗数 Agent** 更进一步，采用 10 步强制推理（含亮度自检步骤），将推理链条形式化程度推得更高。

#### （3）知识注入 — 42 个核心概念的精确定义（vs 最初 31 个）

System Prompt 包含一个完整的"核心概念库"，对 42 个命理学术语给出了严格的操作性定义。例如：

```markdown
**14. 身强**
- 严格定义：日主综合得令、得地、得势后力量充足，能胜任财官
- 区分要点：身强宜克泄耗（官杀、食伤、财星），忌生扶（比劫、印星）
- 实战示例：甲木生于寅月（得令），地支又见卯木（得地），
  天干有比劫（得势），身强无疑
```

每个概念包含：**严格定义 → 区分要点 → 实战示例**。这种结构化知识表示确保了模型在推理时使用统一的术语体系，避免概念混淆导致的推理错误。

#### （4）知识库外挂（knowledge_base/）

早期方案将全部领域知识内联在 System Prompt 中，导致 prompt 臃肿且维护困难。2026 年 6 月参照 [ShenSuan](https://github.com/BIG-CR/ShenSuan) 项目的知识结构化思路重构：

| 文件 | 条数 | 用途 |
|------|------|------|
| `tiaohou.json` | 120 | 穷通宝鉴调候用神表 |
| `signal_rules.json` | 7 数据块 | 冲合/三合/驿马/墓库信号规则 |
| `shishen_domains.json` | 40 | 十神×年龄→人生领域映射 |
| `geju_rules.json` | ~30 | 格局判定+十神组合→职业/婚姻/财运 |
| `error_patterns.json` | 13 | 错误模式结构化定义 |
| `glossary.json` | 44 | 术语词典（前端→服务端统一管理） |
| `classical_references.json` | 5 | 古籍引用→表单 tooltip |
| 紫微文件 8 个 | — | 星曜/宫位/四化/辅佐/格局等 |

`analysis_service.py` 通过 `_load_json_kb()` 按需加载并内存缓存，替代了原先内联 Python dict 的数据管理方式。**效果**：System Prompt 从 ~54KB 瘦身（纯推理指令），知识数据独立维护。

#### （5）约束边界 — 负面清单 + 13 条错误模式

```
### 严格禁止的行为
- 不妄称可以靠某种方法百分百改变命运
- 不推销任何付费化解法事或风水物品
- 不妄断生死精确日期
- 不对用户做道德审判
```

此外，agent 定义中嵌入了 13 条错误模式（盲测驱动发现），作为反面示例注入：

| # | 模式 | 发现来源 |
|---|------|----------|

| 2 | 旺衰层次混淆 | 一致性检查 |
| 3 | 驿马应期宽泛 | 用户验盘纠正 |
| 4 | 均衡命局泛化预测 | 盲测（王菲/林青霞） |
| 5 | 学历时代校正 | 盲测（成龙/林青霞） |
| 6 | 名人识别污染 | 盲测（蒋介石/毛泽东） |
| 7 | 跳过验盘流程 | 盲测（毛泽东） |
| 8 | 国际命例学历偏差 | 盲测（科比） |
| 9 | 特征型→时间型切换 | 盲测（张艺谋重测） |

#### （6）结构化输出格式

System Prompt 规定了标准批命格式（5 个部分），其中"事业财运"和"婚姻感情"各有 6 个必输出子项。这种输出规范保证了分析报告的完整性和一致性。

### 3.2 用户消息构造（Message Construction）

`analysis_service.py` 中的 `_build_user_message()` 函数将排盘数据转换为结构化的 Markdown 分析请求：

```
┌─────────────────────────────────────┐
│ 1. 基础信息（公历/农历/性别/经纬度）  │
│ 2. 四柱干支表（含十神/纳音/藏干/长生）│
│ 3. 核心数据（日主/起运/空亡/胎元）   │
│ 4. 大运排盘表（8步完整表格）         │
│ 5. 流年对照表（双模式：极端→信号列   │
│    均衡→十神🆕首现列）               │
│ 6. 9 章分析要求 + 排版格式要求        │
└─────────────────────────────────────┘
```

**关键设计决策**：
- 使用 Markdown 表格而非 JSON 传递命盘数据，因为 LLM 对 Markdown 表格的理解能力远优于 JSON
- 分析要求明确到章节级别，约束输出结构
- 单次分析 `max_tokens: 24576`，`temperature: 0.3`（需要严谨推理，不需要创意发散）
- 续接对话 `max_tokens: 16384`，降 temperature 0.3

### 3.3 双模式验盘策略（2026-06/07 关键突破）

系统根据命盘"极端度"动态选择验证策略，这是本系统在"让 LLM 做它擅长的事"方面的关键设计：

```
plate_dict → compute_spread()
  ├─ spread ≥ 3 → 冲合信号表（极端命局）
  │    ├─ S级：天克地冲/日柱伏吟（±0年）
  │    ├─ A级：日柱伏吟（±0年）
  │    ├─ B级：六冲日/月/年支（±1年）
  │    ├─ C级：三合/六合（±1年）
  │    ├─ D级：驿马到位（±2年）
  │    ├─ E级：墓库开闭（±2年）
  │    └─ 兜底：大运交接年（±1年）
  │
  └─ spread ≤ 2 → 十神到位法 → 大运主题确认（均衡命局）
       ├─ 查🆕首现（原局没有的十神首次出现）
       ├─ 年龄映射（十神×年龄段映射人生领域）
       └─ 诚实降级：候选 < 3 → 大运主题确认（±2年容差）
```

**盲测数据**：

| 命局类型 | spread | 命中率 | 阶段 |
|----------|--------|--------|------|
| 极端（冲合信号） | ≥3 | 100% | 已验证 |
| 中度 | 2 | 78% | 已验证 |
| 轻度 | 1 | 38% → 71% | 均衡突破后 |
| 均衡 | 0 | 0% → 50-71% | 十神到位法+大运主题确认 |

### 3.4 三通道 SSE 流式管道（2026-06）

参考 GPT-5.5 Thinking 泄露的系统提示词揭示的三层输出通道设计，实现了 SSE（Server-Sent Events）流式分析管道：

```
three_channel.py（723 行）
│
├─ Channel 1 — analysis（主分析通道）
│    完整分析报告，逐段输出
│
├─ Channel 2 — commentary（推理过程）
│    LLM 的"隐藏思考层"，展示每步推理依据
│
└─ Channel 3 — final（精炼输出）
│    对主分析的精简总结，适合终态展示
│
└─ 14 阶段进度事件 → 前端进度条展示
```

**设计价值**：
- 解决 4-6 分钟同步等待的 UX 问题
- 隐藏推理层让用户看到"模型怎么想的"，提升可解释性
- 传统上需要多次 API 调用的"分步输出"，通过单次 SSE 连接完成

### 3.5 多轮对话管理（Conversation Continuation）

系统支持"续接分析"——当 Agent 在验盘阶段需要用户确认过去事件时，用户可以回复，系统会携带完整 `messages` 数组继续对话：

```python
# 初始分析
messages = [
    {"role": "system", "content": system_prompt},
    {"role": "user", "content": user_message},
    {"role": "assistant", "content": analysis_text},
]

# 续接
api_messages = messages[1:]  # 历史对话（不含 system）
api_messages.append({"role": "user", "content": user_reply})
# system prompt 单独传递
payload = {"system": messages[0]["content"], "messages": api_messages}
```

这遵循了 Anthropic Messages API 的 conversation 范式，system prompt 在每轮中保持一致。

### 3.6 API 管理与限流

| 机制 | 实现 |
|------|------|
| **配置优先级** | 环境变量 → `~/.claude/settings.json` → `config.local.py`（适配不同部署环境） |
| **限流** | IP 级别滑动窗口，深度分析 3 次/时，续接 30 次/时/对话（缓存命中不消耗配额） |
| **超时** | 单次分析 480 秒（DeepSeek ~50tok/s），续接对话 480 秒 |
| **Token 追踪** | 返回 `usage` 对象（prompt_tokens / completion_tokens） |
| **密码保护** | 可选 `WEB_PASSWORD` 配置，输错 5 次锁定 3 分钟防爆破 |
| **分析缓存** | 服务端基于四柱+性别+起运生成缓存键，最大 50 条，1 小时 TTL |

### 3.7 验盘截停 + 后处理硬校验（2026-06 新增）

**问题**：LLM 分析完成后，其输出中的硬事实（如起运年龄、大运干支）可能与排盘引擎计算结果不一致。

**旧方案**：双采样自一致性（Pass1 temp=0.3 + Pass2 temp=0.7）→ 成本 ¥0.03/次，UX 差（用户手动对比两段文本），拦截率不确定。

**新方案**：
1. `stop_sequences=['【验盘完毕】']` — Agent 验盘完物理截停，避免模型在验盘后继续生成无关内容
2. `_verify_predictions(analysis_text, plate_dict, current_year)` — 后处理硬校验，用排盘引擎的确定性输出 vs LLM 输出中的事实性断言
3. 成本：¥0/次（纯 Python），可靠性：排盘引擎硬事实 vs LLM 交叉

### 3.8 反馈日志与验盘系统

每次深度分析自动保存到 `feedback/`（被 `.gitignore` 忽略）：

```
feedback/
├── 20260615_034827_乙_8592.json   # 首次分析（含 verification 字段）
```

每份 JSON 含 `verification` 字段（用户通过前端面板提交后写入）：

**用途**：积累标注数据 → `evaluate_agent.py verify-report` 统计命中率 → 提取错误模式 → 更新 Agent few-shot。

## 四、符号计算层（传统 AI 方法）

在 LLM 兴起之前，八字排盘只能通过专家系统实现。本系统的符号计算层本质是一个**确定性规则引擎**。

### 4.1 双引擎策略（鲁棒性设计）

```
优先级 1：sxtwl（C++ 寿星天文历）
    ├─ 精度：±1 秒
    └─ 条件：需编译安装

优先级 2：纯 Python 回退（自动启用）
    ├─ 节气：Meeus 天文公式计算太阳视黄经
    ├─ 精度：~0.001°（起运差异 ~0.1 岁）
    ├─ 优化：3年×24节气预计算缓存（首次后 5000→72 次计算）
    ├─ 依赖：仅 math 模块（零外部依赖）
    └─ 包含：从 solar 到 lunar 的完整回退实现
```

### 4.2 核心算法（八字）

| 算法 | 方法 | 形式化程度 |
|------|------|-----------|
| 四柱干支 | 查表法 + 五虎遁/五鼠遁口诀编码 | 完全形式化 |
| 十神关系 | 五行生克矩阵 + 阴阳异同判定 | 完全形式化 |
| 起运时间 | 节气日差 ÷ 3 = 起运年龄 | 完全形式化 |

| 夜子时处理 | 23:00-23:59 日柱按次日、时柱按次日日干 | 完全形式化 |

### 4.3 紫微斗数算法扩展（2026-07 新增）

紫微斗数安星法（`ziwei_calculator.py`，694 行）是比八字排盘更复杂的符号计算系统：

| 算法 | 方法 | 复杂度 |
|------|------|--------|
| 安星诀 | 年柱→紫微星定位，月柱→紫微系分布 | 14 主星逐步推导 |
| 宫位系统 | 命宫/兄弟/夫妻/子女/财帛/疾厄/迁移等 12 宫 | 生月+生时定位 |
| 四化飞星 | 化禄/化权/化科/化忌（年干+年支定位） | 4 套规则 |
| 亮度计算 | 庙旺得利平不陷（7 级） | 14×12 地支查找表 |
| 特殊格局 | 杀破狼/机月同梁/武贪/紫府等 24 种 | 模式匹配 |
| 身宫定位 | 生年支→身宫宫位（6 种模式） | 条件分支 |
| 对宫冲照 | 6 组对宫力分析 + 7 级冲照力度 | 矩阵运算 |

**亮度误差修正**：iztro-py 引擎的星曜亮度表存在系统偏差。`ziwei_calculator.py` 中集成了 `_BRIGHTNESS_FIX` 全量修正表（14 星×12 地支 = 168 条修正规则，参考令东来亮度表），确保后端返回的亮度数据与主流紫微排盘软件一致。

### 4.4 与 LLM 层的分工边界

**符号计算层负责**：所有可形式化、可查表、可公式化的计算。
**LLM 层负责**：需要模糊判断、经验推理、语境理解的"断命"环节。

这种分工体现了 AI 系统的经典设计原则：**让确定性算法做确定性的事，让概率模型做概率性的事。**

## 五、关键技术挑战与解决方案

### 5.1 LLM 推理一致性问题

**挑战**：同一命盘多次分析可能得出不同结论（这是 LLM 概率性的本质）。

**解决方案**：
- `temperature: 0.3` — 最小化随机性
- 9 级递进推理链 — 约束推理路径
- 42 个概念精确定义 — 统一术语体系
- 必输出子项清单 — 防止遗漏关键维度
- **服务端分析缓存** — 同命盘命中缓存后秒返，避免重复分析导致结论不一致

### 5.2 领域知识注入效率

**挑战**：如何把 30 年命理师经验高效注入 LLM？

**解决方案**：
- 不使用 Few-shot（示例会占据大量 token 且覆盖不足）
- 使用"方法论注入"——教模型"怎么分析"而非"分析什么结论"
- 每个概念用"定义 + 区分 + 示例"三元组表达，兼顾准确性和可操作性
- System Prompt 达 ~123KB，提供足够丰富的领域知识上下文
- **知识库外挂**：将结构化数据（调候用神表、信号规则、十神映射等）分离到 16 个 JSON 文件，按需加载，避免内联臃肿

### 5.3 幻觉抑制

**挑战**：LLM 可能编造不存在的古籍引用或杜撰命理术语。

**解决方案**：
- System Prompt 明确要求"引用经典出处"
- 要求"不确定处标明【待验证】"
- 验盘机制：先反推已知事件验证命盘准确性，再批断未来
- **13 条错误模式注入**：将盲测中发现的典型错误写入 Prompt 作为反面教材
- **三层验证**：查表 → 确认 → 拒答。表中找不到对应关系 → 输出【无法确定】，禁止猜测
- **后处理硬校验**：`_verify_predictions()` 用排盘引擎的确定性输出交叉检查 LLM 输出中的事实性断言

### 5.4 均衡命局分析难题

**挑战**：无明显冲合信号（spread ≤ 2）的均衡命局，LLM 无从下手。2026-06-16 盲测数据：spread=0 的均衡命局命中率为 0%。

**解决路径**（2026-06-29 → 2026-07-13）：

1. **十神到位法**：不找冲合异常，找十神"从无到有"——原局没有的十神在大运/流年首次出现 = 新人生领域启动
2. **诚实降级**：首现候选 < 3 条 → 告知信号不足，降级为大运主题确认
3. **均衡命局信息论上限**：十天干每 10 年轮一圈，任何十神 10 岁前必以天干出现，流年级时间信号在均衡命局中理论上不存在

**最终命中率**：均衡命局从 0% → 50-71%（大运交接年 ±2 年容差）。

### 5.5 长耗时分析的断线恢复

**挑战**：深度分析耗时 4-6 分钟，期间用户切标签页或关闭页面会导致 HTTP 请求中断。

**解决方案（三层防护）**：

| 层级 | 机制 | 说明 |
|------|------|------|
| **服务端缓存** | 四柱+性别+起运作哈希键，50 条/1 小时 TTL | 命中后秒返，不消耗限流配额 |
| **客户端 pending** | `bazi_analysis_pending` 存入 localStorage | 页面加载时检测未过期 pending 自动重试 |
| **运行时恢复** | `visibilitychange` + AbortError 自动重试 | 切回标签页自动恢复，指数退避（5s/15s/30s，3 次） |

此外，分析开始前设置 `beforeunload` 事件提醒用户，8 分钟超时通过 AbortController 自动取消。密码不存 localStorage，重试时需重新输入。

### 5.6 流式输出长耗时问题（2026-06 解决）

最初为同步返回，4-6 分钟等待体验差。通过实现三通道 SSE 流式管道解决：

- 第一个 token 返回时间从 4-6 分钟降到 ~30 秒
- 持续进度推送让用户知道系统"仍在工作"
- 隐藏推理层提升了 UX 和可解释性

### 5.7 Agent 定义幻觉污染（2026-07）

**问题**：Agent MD 中残留了 11.6KB 记忆系统指令（`<read_file>` 等工具调用指令），导致 API 调用时 Agent 输出文件阅读指令。王菲盲测从 60% 跌至 0%。

**解决**：删除 Agent 定义中所有工具调用指令。Agent 定义只包含领域知识 + 推理链 + 约束规则，不包含任何系统/工具操作指令。

### 5.8 部署环境适配

**挑战**：PythonAnywhere 免费账户无环境变量、无法下载文件、WSGI 进程管理特殊、Python 3.11 vs Bash 3.13 不一致。

**解决方案**：多级配置回退（环境变量 → `settings.json` → `config.local.py`），静态资源本地化，禁止 CSP 头和外部 CDN。`pip install` 必须用 `python3.11 -m pip --user`。唯一可靠重启方式为 Disable → 等几秒 → Enable（Reload 不够）。

### 5.9 跨平台中文字体兼容

**挑战**：PDF 生成需要 CJK 字体，但 Linux/macOS/Windows 字体路径各不相同。

**解决方案**：`generate_bazi_pdf.py` 中实现 `_find_cjk_font()` 函数，按优先级自动探测各平台字体路径，并内置 `fonts/` 目录作为回退。

## 十、不足与改进方向

1. **推理可解释性**：虽有三通道流式展示推理过程，但 LLM 分析仍为黑盒输出，可引入"逐层中间结果透出"让用户看到每层推理的确认过程
2. **知识库 RAG**：当前知识库 JSON 是预结构化数据，可将古籍原文（《渊海子平》《滴天髓》等）向量化，分析时检索相关段落作为实时引用依据
3. **一致性验证自动化**：缺少对同一命盘多次分析结论一致性的自动化评估套件
4. **Agent 自进化**：验盘反馈（`feedback/`）数据可用于微调或自动优化 System Prompt
5. **评估基准扩展**：当前盲测覆盖 16+ 名人命例，需扩展到 50+ 命例建立系统化基准
6. **缓存策略优化**：当前为简单 LRU + TTL，可引入持久化缓存（SQLite/Redis），跨进程重启保留
7. **图表交互性**：当前 SVG 为静态渲染，可增加鼠标悬浮高亮、点击展开等交互
8. **新前端进度**：`frontend/` 目录 React 重构正在进行中，需完成渐进式迁移
9. **跨系统数据共享**：八字分析发现（如旺衰结论）目前未自动传递给紫微斗数分析，反之亦然。双命理体系的交叉验证和综合报告是下一步方向

## 十二、总结

本项目实践了"**传统领域知识 × 大语言模型**"的混合 AI 架构。核心创新在于：

1. **用确定性算法（符号计算层）保证底层数据的准确性**——八字排盘不允许有误差
2. **用精心设计的 System Prompt（~123KB）将领域专家的推理方法论注入 LLM**——不是教模型"答案是什么"，而是教它"怎么思考"
3. **用 9 级递进推理链约束模型的推理路径**——强制 CoT，避免跳步推理导致的逻辑混乱
4. **用知识库外挂（16 个 JSON，176KB）替代内联 dict**——结构化数据独立维护，System Prompt 保持推理指令纯度
5. **用双模式验盘策略适配不同命局类型**——极端命局找冲合信号（命中率 100%），均衡命局用十神到位法（命中率 50-71%）
6. **用验盘截停 + 后处理硬校验替代双采样**——¥0/次 的纯 Python 方案，比 ¥0.03/次的 LLM 双采样更可靠
7. **用三通道 SSE 流式管道解决长耗时问题**——从 4-6 分钟同步等待到 ~30 秒首 token + 持续进度推送
8. **用验盘反馈日志形成人机协作闭环**——模型反推过去事件 → 用户验证 → 反馈后模型调整 → 累计标注数据驱动 Agent 迭代
9. **用盲测框架量化评估 Agent 质量**——16+ 名人命例盲测，发现错误模式 → 提取 13 条反例 → 注入 Agent → 再盲测验证的迭代循环
10. **用三层断线恢复机制提升长耗时分析的可靠性**——服务端缓存 + 客户端持久化 + 运行时自动重试，确保 4-6 分钟分析不丢失

这为"AI + 传统文化"领域提供了一个可复用的技术范式：**符号计算做"硬"的部分（可形式化的规则），LLM 做"软"的部分（需要经验判断的模糊推理），两者通过结构化数据接口耦合。**
